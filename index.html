Software Architecture — Exam Notes 7th Sem

Q1: Explain the evolution of software architecture. Discuss how architecture has changed from monolithic systems to modern distributed systems.

Summary: The journey from Monoliths (everything in one box) to Distributed Systems (Microservices, Serverless). Evolution is driven by the need for higher scalability, fault tolerance, and faster deployment speeds (DevOps).

   [ MONOLITHIC ]           [ LAYERED ]             [ SOA ]              [ MICROSERVICES ]
    ___________            ___________            ___________            _______________
   |  _______  |          |____UI_____|          | Service A |          |  [S1]   [S2]  |
   | | Code  | |   -->    |___Logic___|   -->    |___|___|___|   -->    |  (DB)   (DB)  |
   | |_______| |          |___Data____|          |   ESB     |          | [S3]   [S4]   |
   |___|___|___|                                 |___________|          |_______________|
      (1 Unit)             (Separation)          (Integration)           (Decentralized)
Tip: Draw 4 boxes left-to-right. 1. Big box (Monolith). 2. Stack of 3 rectangles (Layered). 3. Services on a bus (SOA). 4. Floating independent boxes (Microservices).

1. The Era of Monoliths (Early Days):
Software started as a single executable. Mainframe systems.
- Structure: UI, Business Logic, and Data Access code were tightly coupled in one codebase.
- Pros: Easy to develop initially, simple deployment (copy 1 file), zero network latency internally.
- Cons: "Spaghetti Code", Single Point of Failure (one bug crashes app), Vertical Scaling only (buy bigger RAM).

2. Client-Server & Layered (1990s):
The rise of the PC led to separating the Client (UI) from the Server (Database).
- 2-Tier: Fat Client -> DB.
- 3-Tier: UI -> Application Server (Logic) -> Database. This introduced logical separation of concerns.

3. Service-Oriented Architecture (SOA) (2000s):
Enterprise-scale integration. Big monolithic apps needed to talk to each other.
- Key Tech: SOAP (XML), ESB (Enterprise Service Bus).
- Focus: Reusability of services across the enterprise. Heavyweight communication.

4. Microservices & Cloud (2010s+):
Driven by Netflix, Amazon, and Cloud (AWS).
- Concept: Break app into tiny, independent services organized around business capabilities (e.g., "Payment Service").
- Key Tech: REST (JSON), Docker containers, Kubernetes.
- Pros: Horizontal scaling, Fault isolation, Tech stack freedom (Java for one, Node.js for another).

FeatureMonolithMicroservices
ScalingVertical (Bigger Hardware)Horizontal (More Machines)
DatabaseShared Central DBDatabase per Service
CouplingTightLoose
DeploymentRisky (Deploy All)Safe (Deploy One Service)

Sample Answer (7 Marks)

The evolution of software architecture reflects the industry's move from centralized, rigid structures to decentralized, flexible systems. The phases are:

1. Monolithic Architecture: The starting point where all components (UI, Logic, Data) reside in a single artifact. It provided simplicity but suffered from tight coupling and a single point of failure. Scaling required upgrading hardware (Vertical Scaling).

2. Client-Server/Layered: Introduced separation of concerns (3-Tier Architecture). It improved maintainability by decoupling the UI from logic and data, but still relied on a centralized database.

3. Service-Oriented Architecture (SOA): Focused on integrating large enterprise applications using an Enterprise Service Bus (ESB). Services communicated via SOAP, emphasizing reusability.

4. Microservices Architecture: The modern standard. Applications are decomposed into small, autonomous services that communicate via lightweight APIs (REST). This enables DevOps, independent deployment, fault isolation, and infinite horizontal scaling on cloud infrastructure.

Mnemonic: M-C-S-M -> Monolith, Client-Server, SOA, Microservices. ("My Computer Saw Me").

Cheat Sheet

  • Monolith: Single point of failure, Vertical Scaling.
  • Client-Server: 2-Tier/3-Tier separation.
  • SOA: ESB, SOAP, Heavyweight integration.
  • Microservices: Decentralized DB, REST, Horizontal Scaling.
  • Trend: Tight Coupling -> Loose Coupling.

Q2: Describe different software development methodologies and the software quality model.

Summary: Methodologies control how we build (Waterfall vs Agile). Quality Models measure how good it is (ISO 9126). Key contrast: Waterfall is linear; Agile is cyclic.

   WATERFALL (Linear)           AGILE (Cyclic)
   [ Requirements ]            (   PLAN   )
          |                  /              \
      [ Design ]           ( DEPLOY      CODE )
          |                 \              /
       [ Code ]              (    TEST    )
          |                   -------------
       [ Test ]               Repeats every
          |                  Sprint (2 weeks)
     [ Maintenance ]
Tip: Draw steps going down like a staircase for Waterfall. Draw a circular cycle for Agile with arrows.

1. Development Methodologies:

  • Waterfall Model: Classical, linear approach. You cannot go back. Requirements must be frozen before design starts.
    Use when: Requirements are clear and fixed (e.g., building a bridge).
    Failures: If requirements change late, the project fails.
  • Agile (Scrum/Kanban): Iterative approach. Software is built in small chunks called "Sprints" (usually 2 weeks).
    Use when: Requirements are unclear or changing (e.g., startup app).
    Key: Continuous feedback from the customer.
  • Spiral Model: Risk-driven. Combines Waterfall phases with prototyping loops. Good for high-risk projects.

2. Software Quality Model (ISO 9126):
The global standard for "What makes software good?". It has 6 top-level attributes:

Sample Answer (7 Marks)

Software Methodologies:
1. Waterfall: A sequential design process where progress flows steadily downwards through Requirement, Design, Implementation, Verification, and Maintenance phases. It is rigid and best for stable requirements.
2. Agile: An iterative methodology that values customer collaboration and responsiveness to change. Development happens in short cycles (Sprints), delivering functional software frequently.
Software Quality Model (ISO 9126):
This standard evaluates software quality based on six characteristics:
1. Functionality: The ability to provide required functions (Accuracy, Security).
2. Reliability: Capability to maintain performance level (Fault tolerance).
3. Usability: Effort needed to learn and use the software.
4. Efficiency: Performance relative to resources used (Speed, Memory).
5. Maintainability: Effort needed to modify or fix the system.
6. Portability: Ability to be transferred from one environment to another.

Mnemonic: F-R-U-E-M-P -> Functionality, Reliability, Usability, Efficiency, Maintainability, Portability.
(Imagine a "Frump"y old computer needing quality checks).

Cheat Sheet

  • Waterfall: Linear, No turning back, Document heavy.
  • Agile: Iterative, Feedback driven, Code heavy.
  • ISO 9126: The 6 pillars of quality.
  • Efficiency vs Reliability: Often a trade-off.

The Architecture Business Cycle (ABC) explains the "Context" of architecture.

1. Inputs (Influences on Architect):
- Stakeholders: The people who care. End-users want usability; Management wants low cost; Security team wants encryption. Architect must trade-off these conflicting goals.
- Developing Organization: Conway's Law: "Organizations design systems that copy their communication structure." If you have 3 teams, you'll likely build a 3-part system.
- Technical Environment: Industry trends. If everyone uses Cloud/AI, the architect is pressured to use them too.
- Architect's Experience: We stick to what we know. If the architect knows Java, the system will likely be Java.

2. Outputs:
The Architecture itself, and eventually the running System.

3. The Feedback Loop (Return Cycle):
The system changes the environment that created it.
- Organization: Now possesses new skills (e.g., "We are now a Microservices company").
- Stakeholders: Once they see the system, they want more features.
- Tech Env: A successful system might define a new industry standard.

Sample Answer (7 Marks)

The Software Architecture Business Cycle (ABC) illustrates that architecture is a result of business, social, and technical factors, and it reciprocally influences them.

1. Inputs to the Architect:
The architect does not work in isolation. They are influenced by:
- Stakeholders: Diverse groups (Users, Managers, Testers) with conflicting requirements (e.g., Performance vs. Security).
- Developing Organization: The staff skills, budget, and team structure (Conway's Law) constrain the design.
- Technical Environment: Standard frameworks and tools available at the time.
- Architect's Experience: Past successes and failures bias the current design.

2. The Feedback Loop:
Once the system is built, it affects its environment:
- It adds new capabilities to the organization.
- It changes stakeholder expectations (leading to new requirements).
- It contributes to the technical environment (best practices).

Mnemonic: S-O-T-E -> Stakeholders, Organization, Tech Env, Experience. (Inputs to Architect).

Cheat Sheet

  • Cycle: Influence -> Build -> Feedback.
  • Conway's Law: Team structure = System structure.
  • Inputs: S-O-T-E.
  • Feedback: A built system changes the requirements for the next one.

Q4: Explain Software Quality Models: McCall Model, Boehm Model, ISO Model, FURPS.

Different eras produced different ways to define "Quality".

1. McCall’s Model (1977):
Organized around three perspectives of the product:
- Product Revision (Can I change it?): Maintainability, Flexibility, Testability.
- Product Transition (Can I move it?): Portability, Reusability, Interoperability.
- Product Operation (Does it work?): Correctness, Reliability, Efficiency, Integrity, Usability.

2. Boehm’s Model (1978):
Similar to McCall but adds a hierarchy. Top level is "General Utility".
- Key difference: It explicitly places Maintainability as a primary driver for utility. If you can't maintain it, it's not useful.

3. FURPS (Hewlett-Packard):
Simple, functional model.
- Functionality: Features, Capabilities.
- Usability: Human factors, Aesthetics.
- Reliability: Frequency of failure, Recoverability.
- Performance: Speed, Throughput, Response time.
- Supportability: Testability, Extensibility, Serviceability.

4. ISO 9126:
The consensus standard. It unifies the previous models into 6 categories: Functionality, Reliability, Usability, Efficiency, Maintainability, Portability.

Sample Answer (7 Marks)

1. McCall’s Model: Classifies quality into three dimensions regarding the product's lifecycle: Product Operation (Correctness, Efficiency), Product Revision (Maintainability, Flexibility), and Product Transition (Portability, Reusability).

2. Boehm’s Model: A hierarchical model focusing on "General Utility". It suggests that for a system to be useful, it must be reliable, efficient, and maintainable. It emphasizes maintainability more than McCall.

3. FURPS Model: Developed by HP, defining quality as:
- Functionality (Feature set)
- Usability (UX)
- Reliability (Failures)
- Performance (Speed)
- Supportability (Maintenance)

4. ISO 9126: The international standard that synthesized these models into six characteristics: Functionality, Reliability, Usability, Efficiency, Maintainability, and Portability.

Cheat Sheet

  • McCall = Rev, Trans, Ops.
  • FURPS = F-U-R-P-S.
  • Boehm = Hierarchy of Utility.
  • ISO = The standard that won.

Q5: Explain structural, framework, dynamic, and process models of software architecture.

Architecture is too complex to view from just one angle. We use different "Views" or models.

  • Structural Model (The "Skeleton"):
    Describes the static organization of the system.
    - What: Components, Classes, Servers.
    - Diagrams: Class Diagram, Component Diagram, Deployment Diagram.
  • Dynamic Model (The "Blood Flow"):
    Describes the runtime behavior. How does the system react to input?
    - What: Control flow, Data flow, Event handling.
    - Diagrams: Sequence Diagram, State Chart, Activity Diagram.
  • Process Model (The "Construction"):
    Describes the build and deployment steps.
    - What: IDEs, Build Scripts, CI/CD pipelines, Team assignments.
    - Diagrams: Gantt charts, Build flow diagrams.
  • Framework Model (The "Pattern"):
    Describes the architectural style or reference architecture used.
    - What: "This is an MVC app" or "This is a Pipe-and-Filter system".
    - Usage: Helps new developers understand the high-level rules immediately.

Cheat Sheet

  • Structural: Static parts (Nouns).
  • Dynamic: Behavior/Events (Verbs).
  • Process: Building/Teams.
  • Framework: Patterns/Styles.

Q6: Describe major architectural styles: Data-flow architecture; Pipes and Filters architecture; Call-and-return architecture.

1. PIPE & FILTER:
   [Source] -> (Pipe) -> [Filter A] -> (Pipe) -> [Filter B] -> [Sink]
   (Data moves continuously. Filter A and B work in parallel if possible.)

2. CALL & RETURN:
        [ Main Program ]
           /       \
      (Calls)     (Calls)
       /             \
   [Sub A]         [Sub B]
      \               /
    (Returns)     (Returns)

1. Data-Flow Architecture:
The structure is determined by the flow of data, not the flow of control. There is no "global state". Components are stateless "workers" that transform inputs to outputs.

2. Pipes and Filters (Specific Data-Flow style):
- Filter: Independent processing unit. Reads input stream, processes, writes output stream. Ideally, it doesn't wait for the whole file; it processes chunk-by-chunk.
- Pipe: The connector buffer.
- Example: UNIX shell: `cat logs.txt | grep "ERROR" | sort`.
- Pros: Reusability (mix and match filters), Concurrency.
- Cons: Performance overhead (parsing/unparsing data between filters).

3. Call-and-Return Architecture:
The classic style used in C/Java/C++.
- Main-Subroutine: Hierarchical. Main calls Sub, Sub waits until done, returns control.
- OO Systems: Objects call methods on other objects.
- Pros: Easy to reason about ("I know what line runs next").
- Cons: Tightly coupled. Hard to distribute across networks (requires RPC).

Sample Answer (7 Marks)

1. Data-Flow Architecture:
In this style, the system is viewed as a series of transformations on data. Components (transformers) are connected by data pathways. Availability of data triggers the computation.

2. Pipes and Filters:
A sub-style of Data-Flow.
- Filters: Independent components that process data streams. They do not share state.
- Pipes: Conduits that transfer data.
- Key Feature: Filters are ignorant of who sends or receives data, enabling high reusability (e.g., UNIX commands).

3. Call-and-Return:
The dominant style in imperative programming. Control flows in a hierarchy.
- Main-Subroutine: A main controller invokes specific subroutines.
- Object-Oriented: Objects encapsulate state and interact via method invocations.
- Key Feature: Execution is synchronous and sequential.

Cheat Sheet

  • Pipe & Filter: Stream processing. No shared state.
  • Call & Return: Synchronous. Hierarchy.
  • Data Flow: Data drives the system.

Q7: What are the Architectural Constraints of RESTful API?

Summary: REST isn't just "using JSON". It's a set of 6 strict constraints defined by Roy Fielding. If you break them, it's just an HTTP API, not REST.

The 6 Constraints of REST:

  1. Client-Server: Strict separation. Server stores data; Client handles UI. They can evolve separately.
  2. Stateless (CRITICAL): The server contains no client state. Each request contains all info needed (e.g., Token). Server doesn't remember "User X is logged in".
  3. Cacheable: Responses must define if they can be cached. (e.g., "Images are cached for 1 day"). Reduces server load.
  4. Uniform Interface:
    • Standard Identifiers (URIs).
    • Standard Methods (GET, POST, PUT, DELETE).
    • HATEOAS (Hypermedia as Engine of Application State) - API returns links to next actions.
  5. Layered System: Client doesn't know if it's talking to the DB server, a Load Balancer, or a CDN. Allows adding security/caching layers transparently.
  6. Code on Demand (Optional): Server can send logic (JavaScript) to client to extend functionality.
Mnemonic: S-C-U-L-C-S (Skulls) -> Stateless, Client-Server, Uniform Interface, Layered, Cacheable, Script (Code on demand).

Cheat Sheet

  • Stateless: No sessions on server.
  • Uniform Interface: Standard verbs (GET/POST).
  • Cacheable: Browser saves data.
  • HATEOAS: API provides links.

Q8: Explain layered architecture and data-centered architecture with diagrams.

A. LAYERED STYLE:              B. DATA-CENTERED (Repository):
_____________________              [ Client A ]    [ Client B ]
| Presentation (UI) |                    \            /
|___________________|                     v          v
         | v                          ____________________
_____________________                |                    |
| Business Logic    |                |    SHARED DATA     |
|___________________|                |    REPOSITORY      |
         | v                         |____________________|
_____________________                     ^          ^
| Data Access (DAO) |                    /            \
|___________________|              [ Client C ]    [ Client D ]

1. Layered Architecture:
Organizes the system into horizontal layers of abstraction.
- Rules: Layer N provides services to Layer N+1. Layer N+1 only calls Layer N (Strict layering).
- Pros: Separation of concerns. You can replace the DB layer (Oracle -> Mongo) without touching the UI.
- Cons: Performance overhead (data must pass through every layer, even if logic layer does nothing - "Sinkhole antipattern").

2. Data-Centered (Repository) Architecture:
A central data store (Database, Registry, Blackboard) is the primary means of communication.
- Passive Repository: Clients pull/push data (e.g., Traditional DB apps).
- Active Blackboard: The data store notifies clients when data changes (e.g., AI systems).
- Pros: Data integrity is easy to manage in one place. Clients are decoupled from each other.
- Cons: The Repository is a bottleneck and single point of failure.

Cheat Sheet

  • Layered: UI -> Logic -> Data. Abstraction stack.
  • Data-Centered: Hub and Spoke. Star topology.
  • Use Layered for: Web Apps.
  • Use Data-Centered for: IDEs, CAD tools, AI.

Q9: What is Microservices Architecture? Explain benefits and challenges.

Definition: An architectural style where an application is a collection of small, autonomous services, each running in its own process, communicating via lightweight mechanisms (HTTP), and independently deployable.

Core Principles:
- Database per Service: No shared big database.
- Smart Endpoints, Dumb Pipes: Logic is in the service, not the bus.
- Decentralized Governance: Teams choose their own tools.

Benefits:
1. Agility & Speed: Small teams work in parallel. Faster release cycles.
2. Flexible Scaling: Scale only the "Search" service during high traffic, not the "About Us" page.
3. Technology Freedom: Use the best tool for the job (Node for I/O, Python for AI).
4. Resilience: If "Reviews" service fails, "Checkout" still works.

Challenges:
1. Distributed Complexity: Network latency, partial failures, debugging across services.
2. Data Consistency: No ACID transactions across services. Must use "Eventual Consistency" (Sagas).
3. Operational Overhead: Need complex DevOps (Docker, Kubernetes, Monitoring).

Cheat Sheet

  • Keyword: Independence.
  • Pros: Scale, Speed, Resilience.
  • Cons: Complexity, Network issues, Data sync.
  • Mnemonic: S-I-D -> Small, Independent, Distributed.

Q10: Describe agent-based architecture and reactive architecture.

1. Agent-Based Architecture:
Components are "Agents" - autonomous entities with goals.
- Autonomy: They act without human intervention.
- Social Ability: They negotiate with other agents.
- Reactivity: They perceive environment changes.
- Example: Robotics swarm, Stock trading bots.

2. Reactive Architecture (The Reactive Manifesto):
Systems designed to be robust in the modern cloud era. Defined by 4 traits:
1. Responsive: Respond quickly to users (Latency control).
2. Resilient: Remain responsive even when things fail (Fault tolerance).
3. Elastic: Remain responsive under varying workload (Auto-scaling).
4. Message Driven: The foundation. Use async message passing to achieve the above.

Reactive Traits: R-R-E-M -> Responsive, Resilient, Elastic, Message-Driven.

Q11: Give explanation on event-driven architecture.

Summary: A pattern where components communicate by emitting "Events" (Facts that happened). Decouples producers from consumers.

   [ Producer ] --> (Event: "User Created") --> [ EVENT BROKER ]
                                                     |
                                    -----------------------------------
                                    |                |                |
                                    v                v                v
                               [Consumer A]     [Consumer B]     [Consumer C]
                               (Send Email)     (Create Log)     (Update Analytics)

Concept:
In traditional Request/Response, A calls B (A waits for B).
In Event-Driven, A says "X happened" and forgets it. B, C, and D listen for "X" and react.
- Decoupling: The producer doesn't care who listens.
- Asynchronous: Non-blocking.

Topologies:
1. Mediator Topology: Events go to a central "brain" (Mediator) which decides the steps. Good for complex workflows.
2. Broker Topology: No central brain. Events flow in a chain. Good for simple, high-performance flows.

Q12: Explain Software Architecture Description Languages (ADLs). Discuss types and features.

Definition: Before UML was standard, we used ADLs. These are formal languages to write down architecture precisely.

Essential Features of an ADL:
1. Components: Ability to define computational units.
2. Connectors: Ability to define interactions (pipes, calls).
3. Configurations: Ability to show the graph/topology.
4. Analysis: Ability to mathematically check for deadlocks or speed.

Famous ADLs:
- ACME: The "XML" of architectures. Interchange format.
- Wright: Used for formal analysis of deadlocks.
- Darwin: Used for distributed systems structure.

Q13: Explain CORBA and Java RMI architecture with use cases.

      CLIENT MACHINE                 SERVER MACHINE
   ____________________           ____________________
  |    [ Object ]      |         |    [ Object ]      |
  |        |           |         |        ^           |
  |     (Stub)         |         |    (Skeleton)      |
  |________|___________|         |________|___________|
           |                              ^
           |______________________________|

Both are legacy technologies for Distributed Objects.

1. CORBA (Common Object Request Broker Architecture):
- Goal: Language Independence. A C++ object calls a Java object.
- Key: ORB (Object Request Broker). The middleware bus that translates messages.
- IDL: Interface Definition Language. You write the interface in IDL, and compile it to C++ or Java code.

2. Java RMI (Remote Method Invocation):
- Goal: Java-to-Java communication.
- Mechanism: Uses serialization. The Client has a "Stub" (fake object). When you call the Stub, it sends data over network to the "Skeleton" on server, which calls the real object.
- Advantage: Easy if you only use Java.

MVC (Model-View-Controller):
The standard for interactive applications (Web & Desktop).

Q16: Explain JSP, and lifecycle of servlets.

      [ Load Class ]
            |
        [ Instantiation ]
            |
         [ init() ]  <--- Called ONCE
            |
  -------> [ service() ] <--- Called per Request (Thread)
  |         |
  |______ [ doXxx() ]
            |
        [ destroy() ] <--- Called on Shutdown

1. JSP (JavaServer Pages):
A way to write HTML with embedded Java code (`<% code %>`).
- Process: JSP file -> Translated to Java Source -> Compiled to Servlet Class -> Run.
- Use: View layer (presentation).

2. Servlet Lifecycle:
Servlets are the Java classes that handle HTTP requests. The Container (Tomcat) manages them:
1. Load & Instantiate: Server creates object.
2. init(): Initialization logic (DB connection). Runs ONCE.
3. service(): The main loop. Runs for EVERY request. Dispatches to `doGet`/`doPost`.
4. destroy(): Cleanup logic. Runs ONCE on shutdown.

Mnemonic: L-I-S-D -> Load, Init, Service, Destroy.

Q17: What is the Cost Benefit Analysis Method (CBAM)? Explain the steps.

CBAM answers the question: "We have 5 good architectural ideas, but only budget for 2. Which ones give the most ROI?"
It adds Economic modeling to technical decisions.

Steps:
1. Collate Scenarios: Collect ideas.
2. Refine Scenarios: Filter the best ones.
3. Prioritize Scenarios: Vote on importance.
4. Assign Utility: Give a "Benefit Score".
5. Develop Strategies: How to implement them?
6. Determine Cost: Estimate price/time.
7. Calculate ROI: Benefit divided by Cost. Rank them.

Keywords: ROI, Economics, Budget, Priority.

Q18: Explain Attribute Driven Design (ADD) with an example.

ADD is a design process where the Quality Attributes (Performance, Security) drive the structure.
- Philosophy: Don't just design for features. Design for qualities.

Recursive Steps:
1. Pick a module.
2. Identify top drivers (e.g., "Must handle 10k users").
3. Select a pattern that solves it (e.g., "Load Balancer").
4. Instantiate the pattern.
5. Repeat for children.

Example:
- Goal: High Availability.
- Tactic: Active-Passive Redundancy.
- Design: Add a backup server and heartbeat monitor.

Q19: Describe Architecture Tradeoff Analysis Method (ATAM) with phases and outputs.

ATAM is a review method. We evaluate the architecture before coding to find risks.

4 Phases:
1. Presentation: Present business goals and architecture.
2. Investigation: Analyze approaches and generate Utility Tree.
3. Testing: Brainstorm scenarios (Failures, Growth) and test the design against them.
4. Reporting: Document findings.

Outputs (Mnemonic: R-N-S-T):
- Risks: Bad decisions.
- Non-Risks: Good decisions.
- Sensitivity Points: Where a small change has big impact.
- Trade-off Points: Where improving X hurts Y.

Q20: Describe Active Reviews for Intermediate Design (ARID).

ARID is for reviewing partial designs (e.g., just the subsystem interface).
- The Problem: ATAM is for the whole system. What if I just want to check my API design?
- The Process: Gather the Developers who will use the design. Give them a scenario. Ask them to write pseudo-code using your design. If they can't do it easily, your design fails.
- Hybrid: It's part "Design Review" and part "Code Walkthrough".

Q21: Define Refinement and also explain principle of sound documentation in detail.

Refinement: The gradual process of converting high-level specifications into low-level executable code. (Concept -> Architecture -> Design -> Code).

Sound Documentation Principles:
1. Write for the Reader: Know your audience (Manager vs Developer).
2. Don't Repeat: Have a "Single Source of Truth".
3. Record Rationale: Explain WHY you chose X, not just THAT you chose X.
4. Keep Current: Wrong documentation is dangerous. Update it or delete it.
5. Standardize: Use templates.

Q22: Describe seven parts template for documentation package.

When documenting a View (e.g., Component View), use this standard 7-part structure:

  1. Primary Presentation: The diagram itself (UML).

Q23: What are architecture software interfaces? Explain with examples.

An Interface is a contract. It defines what a component does, hiding how it does it.

Types:
1. Provided Interface (Lollipop): "I can do this for you." (e.g., `PaymentService.process()` )
2. Required Interface (Socket): "I need this to work." (e.g., `PaymentService` needs `DatabaseConnection`).

Importance:
Interfaces allow teams to work in parallel. As long as the interface doesn't change, you can rewrite the internal code completely without breaking the system.
