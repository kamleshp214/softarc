Q1: Evolution of Software Architecture (Monolith to Distributed)

1. Driven by need for Scalability & Agility.
2. Path: Monolith → Client-Server → SOA → Microservices.
3. Shift: Tight Coupling → Loose Coupling.
The evolution of software architecture refers to the historical progression of design patterns from single-unit __Monoliths__ to decentralized __Distributed_Systems__, aiming to solve complexity, scaling, and deployment challenges. 
  [ 1980s ]      [ 1990s ]         [ 2000s ]         [ 2010s+ ]
 MONOLITH     CLIENT-SERVER          SOA           MICROSERVICES
 ________      ___________       ___________      ________________
|        |    |    UI     |     | Service A |    | [S1]   [S2]    |
| All-in |    |___________|     |___|___|___|    |  |      |      |
|  One   |          |             |  ESB  |      | (DB)   (DB)    |
|________|    |___DB______|     |_________|      |________________|
Timeline: 1 Big box → 2 Boxes (UI/DB) → Bus (ESB) → Many floating boxes.

1. Monolithic Architecture (The Beginning): In the early days (Mainframes), software was a single unified block. The UI, Business Logic, and Data Access layers were physically combined in one executable.
Pros: Simple to develop and test initially.
Cons: __Single_Point_of_Failure__ (one bug crashes all), difficult to scale (must buy bigger hardware - Vertical Scaling).

2. Client-Server & Layered (The 90s): The rise of PCs led to splitting the workload. The __Client__ handled presentation, and the __Server__ handled data. This evolved into N-Tier architecture (Presentation, Logic, Data layers).
Improvement: Logical separation of concerns.

3. Service-Oriented Architecture (SOA) (The 2000s): Enterprises needed large, disparate systems to talk to each other. SOA introduced the __Enterprise_Service_Bus_(ESB)__. Services were coarse-grained and communicated via heavy protocols like SOAP/XML.
Focus: Reusability and Integration.

4. Microservices (Modern Distributed): Driven by Cloud (AWS/Azure) and DevOps. Applications are broken into tiny, autonomous services, each with its own database. They communicate via lightweight __REST_APIs__.
Key Benefit: __Horizontal_Scaling__ (add more cheap servers) and Fault Isolation (if one service fails, others survive).

Sample Answer (7 Marks)

The evolution of software architecture has been driven by the need to manage increasing complexity and scale. The major phases are:

  1. Monolithic Architecture: The earliest systems where all modules (UI, Logic, Data) were bundled into a single artifact. While simple, they suffered from tight coupling and vertical scaling limits.
  2. Client-Server/Layered: Introduced logical separation (e.g., 3-Tier Architecture), decoupling the User Interface from the Database.
  3. Service-Oriented Architecture (SOA): Focused on enterprise integration using an ESB and SOAP, allowing large systems to interoperate.
  4. Microservices: The current standard for cloud-native apps. It decomposes systems into small, independent services communicating via REST. This enables __continuous_deployment__ and infinite horizontal scaling.

What to write to get 7/7:

  • Mention "Vertical Scaling" vs "Horizontal Scaling".
  • Keyword: __Coupling__ (Tight to Loose).
  • Mention the timeline: Mainframe -> PC -> Web -> Cloud.
  • Draw the diagram showing the breakdown of the box.
M-C-S-M (Monolith, Client-Server, SOA, Microservices)

Cheat Sheet

  • Monolith: Single executable, hard to change.
  • Client-Server: Separation of UI and Data.
  • SOA: Enterprise Service Bus (ESB), SOAP.
  • Microservices: Independent, REST, Cloud-native.
  • Driver: Need for speed (Time to market) and Scale.

Q2: Development Methodologies & Quality Model (ISO 9126)

1. Waterfall (Linear) vs Agile (Iterative).
2. ISO 9126: The standard for quality.
3. Attributes: Functionality, Reliability, Usability. 
  WATERFALL (Linear)          AGILE (Iterative)
 [ Reqs ]                    (    Plan    )
    |                      /            \
 [ Design ]              ( Deploy      Code )
    |                      \            /
 [ Code ]                    (    Test    )
    |                         ------------
 [ Test ]                    (Repeats Loop)

Part 1: Methodologies
1. Waterfall Model: A sequential design process. Progress flows steadily downwards (like a waterfall) through phases: Requirements -> Design -> Implementation -> Verification -> Maintenance.
- Pros: Simple, disciplined. Good for fixed-requirement projects (e.g., construction).
- Cons: Rigid. __No_turning_back__. Poor for changing requirements.

2. Agile Methodology (Scrum/Kanban): An iterative approach. Software is developed in small increments called __Sprints__ (usually 2 weeks).
- Pros: Adaptable to change, frequent delivery, customer feedback loop.
- Cons: Scope creep, requires experienced team.

Part 2: Software Quality Model (ISO 9126)
Defines "Quality" using 6 standard characteristics:

  • Functionality: Does it do what is needed? (Suitability, Security).
  • Reliability: Does it stay up? (Fault tolerance, Recoverability).
  • Usability: Is it easy to use? (Learnability).
  • Efficiency: Performance relative to resources (Time behavior, Memory).
  • Maintainability: Easy to fix? (Testability, Stability).
  • Portability: Easy to move? (Installability).

Sample Answer (7 Marks)

Software Development Methodologies:
1. Waterfall: A classical linear model where each phase must be completed before the next begins. It relies on heavy documentation and fixed requirements.
2. Agile: A flexible, iterative approach focusing on rapid delivery of small software increments. It values response to change over following a plan.

Software Quality Model (ISO 9126):
To objectively measure software quality, we use the ISO 9126 standard which defines six characteristics:
- Functionality: Suitability and accuracy.
- Reliability: Maturity and fault tolerance.
- Usability: Ease of understanding.
- Efficiency: Resource utilization and speed.
- Maintainability: Ease of modification.
- Portability: Adaptability to environments.

F-R-U-E-M-P (Functionality, Reliability, Usability, Efficiency, Maintainability, Portability)

Q3: Architecture Business Cycle (ABC)

1. Architecture is a cycle, not a one-time event.
2. Inputs: Stakeholders, Org, Tech.
3. Feedback: System changes the environment. 
  [Stakeholders]   [Organization]   [Tech Env]
        |                |               |
        | (Reqs)         | (Skills)      | (Tools)
        v                v               v
     _________________________________________
    |           THE ARCHITECT                 |
    |_________________________________________|
                     |
                     v
              [ ARCHITECTURE ]
                     |
                     v
                [ SYSTEM ]
               /     |    \
(Feedback)    v      v     v
    [Stakeholders] [Org] [Tech Env]

Definition: The __Architecture_Business_Cycle_(ABC)__ is the concept that software architecture is a result of technical, business, and social influences, and the resulting system in turn influences those very factors.

1. Influences on the Architect (Inputs):
- Stakeholders: Users, managers, and testers all have conflicting goals (e.g., "Fast" vs "Secure" vs "Cheap"). The architect mediates these.
- Developing Organization: __Conway's_Law__ states that systems mirror the communication structure of the team. If you have 3 teams, you get a 3-part compiler.
- Technical Environment: Current industry standards (e.g., "Everyone uses Cloud now") force design choices.
- Architect's Experience: We stick to what we know. Past success biases future design.

2. The Feedback Loop (Return Cycle):
Once the system is built, it changes the world:
- Organization: Acquires new skills (e.g., "Now we know Microservices").
- Stakeholders: Seeing the system generates NEW requirements.
- Tech Env: A successful system might become a platform for others.

Sample Answer (7 Marks)

The Software Architecture Business Cycle (ABC) demonstrates that architecture is not created in a vacuum. It is a loop:

Phase 1: Influences (Inputs)
The architect designs the system based on constraints from:
- Stakeholders: Who define the functional and non-functional requirements.
- Developing Organization: Staff skills and budget (Conway's Law).
- Technical Environment: Available tools and frameworks.
- Architect’s Experience: Personal bias and expertise.

Phase 2: Output
These inputs produce the __Architecture__, which leads to the running __System__.

Phase 3: Feedback
The existence of the system changes the environment. It creates new market opportunities, trains the organization in new technologies, and sets new standards, thus restarting the cycle.

S-O-T-E (Stakeholders, Org, Tech, Experience)

Q4: Quality Models (McCall, Boehm, FURPS)

1. McCall’s Model (1977):
Focuses on three perspectives of the software lifecycle:
- Product Revision (Can I change it?): Maintainability, Flexibility, Testability.
- Product Transition (Can I move it?): Portability, Reusability, Interoperability.
- Product Operation (Does it run?): Correctness, Reliability, Efficiency, Integrity, Usability.

2. Boehm’s Model (1978):
A hierarchical model focusing on "General Utility".
- Unique feature: It places __Maintainability__ as a primary driver. If software costs too much to maintain, its utility is low.
- Hierarchy: Utility -> Portability, As-is Utility, Maintainability.

3. FURPS Model (HP):
A simplified industry model.
- Functionality (Features, Security)
- Usability (Human factors)
- Reliability (Frequency of failure)
- Performance (Response time, Throughput)
- Supportability (Extensibility, Serviceability)

What to write to get 7/7:

  • List all 4 models clearly.
  • For McCall, mention the 3 "Product" categories.
  • For FURPS, spell out the acronym.
  • Mention that ISO 9126 is the international standard.

Q5: Architectural Models (Views)

Definition: These models represent different __Viewpoints__ of a system. No single model can describe an entire architecture.

1. Structural Model (The Skeleton):
Describes the static organization of the system. "What is it made of?"
- Components: Classes, Objects, Modules, Servers.
- Diagrams: Class Diagrams, Component Diagrams.
- Use: For developers to understand the code structure.

2. Dynamic Model (The Blood Flow):
Describes the runtime behavior. "What happens when I click this?"
- Elements: State transitions, Message passing, Data flow.
- Diagrams: Sequence Diagrams, State Charts.
- Use: For debugging and performance analysis.

3. Process Model (The Construction):
Describes the steps to build and run the system. "Who does what?"
- Elements: IDEs, Build scripts, CI/CD pipelines, Team assignments.
- Use: For project managers and DevOps.

4. Framework Model (The Pattern):
Describes the high-level architectural style or reference architecture.
- Examples: "This is an MVC app", "This is a Pipe-and-Filter system".
- Use: For quick understanding of the system's philosophy.

Sample Answer (7 Marks)

Software architecture is described using multiple models (views):

  1. Structural Model: Focuses on the static components (modules, classes, databases) and their relationships. It answers "What is the system composed of?".
  2. Dynamic Model: Focuses on the behavioral aspect. It describes how the system changes state and how components interact during execution (e.g., sequence of events).
  3. Process Model: Focuses on the construction steps, deployment processes, and concurrency. It answers "How is the system built and run?".
  4. Framework Model: Refers to the skeletal structures or patterns (like MVC or CORBA) used to solve specific problems, providing a template for the architecture.

Q6: Styles: Data-Flow, Pipe & Filter, Call & Return

1. Data-Flow Architecture:
In this style, the entire structure is determined by the motion of data. Components are "workers" that wait for input data, transform it, and output it. There is __no_global_state__.

2. Pipes and Filters (Subtype of Data-Flow):
- Filters: Independent processing units. They read a stream, process it incrementally, and write to output. They do NOT know identity of upstream/downstream filters.
- Pipes: Connectors that buffer and carry data.
- Example: UNIX Shell (`cat file | grep "err" | sort`).
- Pros: High Reusability (mix & match filters), Concurrency.
- Cons: Data parsing overhead between filters.

3. Call-and-Return Architecture:
The dominant style in classical programming.
- Main-Subroutine: A main controller explicitly calls sub-routines. The caller waits for the callee.
- Object-Oriented: Objects encapsulate data and call methods on each other.
- Pros: Easy to reason about (sequential flow).
- Cons: Tightly coupled components. Hard to distribute.

PIPE & FILTER:
 [Source] -> (Pipe) -> [Filter A] -> (Pipe) -> [Filter B] -> [Sink]

CALL & RETURN:
      [ Main Program ]
      /      |       \
   (Call)  (Call)   (Call)
    /        |         \
 [Sub A]   [Sub B]    [Sub C]
    \________|_________/
         (Returns)

Sample Answer (7 Marks)

1. Data-Flow Architecture: Access and movement of data define the system. Availability of data triggers computation. It promotes loose coupling.

2. Pipes and Filters Style: A specific data-flow style where:
- Filters: Are independent components that transform data streams.
- Pipes: Are conduits connecting filters.
- It is highly flexible; filters can be rearranged easily (e.g., Unix commands). However, it suffers from data parsing overhead.

3. Call-and-Return Style: The classical imperative approach.
- Main-Subroutine: A hierarchy where a main program invokes subroutines. Execution is sequential and synchronous.
- It is simple to understand but scales poorly for distributed applications due to tight coupling.

Q7: Constraints of RESTful API

REST is defined by 6 Constraints. Breaking them means it's not REST.

Definition: REST (Representational State Transfer) is a resource-based architectural style defined by Roy Fielding. To be RESTful, a system MUST adhere to these 6 constraints:

  1. Client-Server: Strict separation of concerns. The server stores data; the client handles UI. They can evolve independently.
  2. __Stateless__ (Critical): The server contains NO client state. Each request must contain all information (e.g., Auth Token) necessary to understand it. The server does not remember "User X is logged in".
  3. Cacheable: Responses must explicitly label themselves as cacheable or non-cacheable. This improves client-side performance.
  4. Uniform Interface: The core constraint.
    • Resource Identification (URIs).
    • Standard Methods (GET, POST, PUT, DELETE).
    • HATEOAS (Hypermedia as Engine of Application State) - API returns links to next actions.
  5. Layered System: The client cannot tell if it is connected directly to the end server or an intermediary (Load Balancer, CDN).
  6. Code on Demand (Optional): The server can extend client functionality by sending executable code (e.g., JavaScript).
S-C-U-L-C-S (Stateless, Client-Server, Uniform, Layered, Cacheable, Script)

What to write to get 7/7:

  • List exactly 6 constraints.
  • Emphasize Statelessness.
  • Mention HATEOAS under Uniform Interface.
  • Mention that "Code on Demand" is optional.

Q8: Layered vs Data-Centered Architecture

A. LAYERED:                  B. DATA-CENTERED:
_____________________          [Client A]  [Client B]
| Presentation (UI) |               \        /
|___________________|                v      v
         | v                       [REPOSITORY]
_____________________              [ DATABASE ]
| Business Logic    |                ^      ^
|___________________|               /        \
         | v                   [Client C]  [Client D]
_____________________
| Data Access (DAO) |
|___________________|

1. Layered Architecture:
Organizes the system into horizontal layers of abstraction.
- Rule: Layer N provides services to Layer N+1. Layer N+1 only calls Layer N (Strict layering).
- Common Layers: Presentation (UI) -> Business Logic -> Data Access (DAO) -> Database.
- Pros: Separation of concerns, Testability, Easy to swap layers (e.g., change DB without touching UI).
- Cons: Performance overhead ("Sinkhole antipattern" - data passing through layers without logic).

2. Data-Centered (Repository) Architecture:
A central data store is the primary means of communication.
- Components: A Central Repository (Database) and independent Clients that access it.
- Passive Repository: Clients pull/push data (e.g., Standard DB apps).
- Active Blackboard: Repository notifies clients when data changes (e.g., AI systems, IDEs).
- Pros: Data integrity is centralized. Clients are decoupled from each other.
- Cons: Repository is a __Single_Point_of_Failure__ and performance bottleneck.

Sample Answer (7 Marks)

1. Layered Architecture: A hierarchical style where components are grouped into layers (Presentation, Business, Data). Each layer serves the one above it.
Benefit: It supports separation of concerns and maintainability.
Drawback: Request latency increases as data traverses multiple layers.

2. Data-Centered Architecture: Relies on a central __Repository__ accessible by all components. Components interact via the data store, not directly with each other.
Benefit: Excellent for data integrity and integrating independent tools (e.g., Eclipse IDE).
Drawback: The repository becomes a performance bottleneck.

Q9: Microservices: Benefits & Challenges

1. Small, independent services.
2. Database per Service.
3. Massive scaling, high complexity.

Definition: An architectural style where an application is a collection of small, autonomous services, each running in its own process, communicating via lightweight mechanisms (HTTP/REST), and independently deployable.

Core Principles:
- Database per Service: No shared big database.
- Smart Endpoints, Dumb Pipes: Logic is in the service, not the bus.
- Decentralized Governance: Teams choose their own tools.

Benefits:
1. Agility: Small teams can work in parallel. Faster time-to-market.
2. Flexible Scaling: Scale only the "Search" service during high traffic, not the "Login" service.
3. Technology Heterogeneity: Use Python for AI service, Java for Banking service.
4. Fault Isolation: If one service crashes, the rest of the app survives.

Challenges:
1. Distributed Complexity: Network latency, debugging is hard (Tracing).
2. Data Consistency: No ACID transactions across services. Must use "Eventual Consistency" (Sagas).
3. Operational Overhead: Requires DevOps (Docker, Kubernetes).

S-I-D (Small, Independent, Distributed)

Q10: Agent-based & Reactive Architecture

Part 1: Agent-Based Architecture
Composed of "Agents"—autonomous components that perceive their environment and act to achieve goals.
- Characteristics:

  • Autonomy: Acts without direct intervention.
  • Reactivity: Perceives environment changes.
  • Social Ability: Negotiates/Collaborates with other agents.

- Use Cases: Robotics, Stock trading bots, Traffic control systems.

Part 2: Reactive Architecture (The Reactive Manifesto)
Systems designed to be robust in the modern cloud/mobile era. Defined by 4 traits:
1. Responsive: Responds in a timely manner (Low latency).
2. Resilient: Stays responsive even when failures occur (Fault tolerance).
3. Elastic: Stays responsive under varying workload (Auto-scaling).
4. Message Driven: Uses asynchronous message passing to ensure loose coupling (The Foundation).

R-R-E-M (Responsive, Resilient, Elastic, Message-driven)

Q11: Event-Driven Architecture (EDA)

 [ Producer ] ---> (Event: "OrderPlaced")
                         |
                   [ EVENT BROKER ]
                   /      |       \
            (Push)     (Push)     (Push)
             /            |           \
      [Email Svc]    [Stock Svc]   [Log Svc]

Definition: A pattern where components communicate by emitting __Events__ (facts that happened, e.g., "UserClickedBuy") rather than direct calls. Consumers listen for events and react.

Key Concepts:
- Decoupling: The producer does not know who is listening. It just yells "Order Placed!" and moves on.
- Asynchronous: Non-blocking. The producer doesn't wait for the consumer.
- Broker Topology: A central Event Bus (like Kafka/RabbitMQ) manages the flow.

Pros: High scalability, easy to add new features (just add a new listener).
Cons: Hard to trace execution flow (no linear stack trace).

Sample Answer (7 Marks)

Event-Driven Architecture (EDA) is a design pattern that promotes the production, detection, and consumption of events.
Mechanism: Instead of direct requests, components emit events to an Event Bus. Subscribers listen to specific events and react asynchronously.
Benefits: It creates highly decoupled systems. Producers don't need to know about consumers, allowing for easy extensibility and high throughput.
Use Case: Real-time systems like Uber or Stock Markets.

Q12: Architecture Description Languages (ADLs)

Definition: ADLs are formal languages used to describe software architecture precisely, focusing on components and connectors (unlike programming languages which focus on algorithms).

Essential Features of an ADL:
1. Components: Defines units of computation (e.g., database, server).
2. Connectors: Defines interactions (e.g., pipes, SQL link).
3. Configuration: Defines the topology/graph.
4. Tool Support: Allows analysis (deadlock detection) and code generation.

Types/Examples:
- ACME: A generic interchange language to swap data between tools.
- Wright: Focuses on formal analysis of deadlocks and protocols.
- Darwin: Focuses on distributed systems and dynamic structures.

Q13: CORBA & Java RMI

 [ Client Object ]        [ Server Object ]
        |                        ^
     (Stub)                  (Skeleton)
        |________________________|
             NETWORK / ORB

1. CORBA (Common Object Request Broker Architecture):
- Goal: Language Independence. A C++ object calls a Java object.
- Mechanism: Uses an __ORB__ (Object Request Broker) as a middleware bus. Interfaces are defined in IDL (Interface Definition Language).
- Use Case: Legacy banking systems mixing Cobol mainframes with C++ clients.

2. Java RMI (Remote Method Invocation):
- Goal: Java-to-Java communication.
- Mechanism: Uses Stubs (Client-side proxy) and Skeletons (Server-side helper). It uses Java Serialization to send objects over the wire.
- Use Case: Pure Java enterprise applications (EJB) in the early 2000s.

Comparison: CORBA is complex but universal. RMI is simple but Java-only.

Q14: Model View Architecture (MVC)

     User Input
         |
         v
   [ CONTROLLER ] -----------> [ MODEL ]
   (Logic/Handler)            (Data/State)
         |                         |
    Selects View              Updates View
         |                         |
         v                         v
     [ VIEW ] <--------------------|
    (Display UI)

Definition: MVC (Model-View-Controller) is a pattern to separate internal representation of information from the way it is presented to the user.

Components:
1. Model: The Brain. Contains data, state, and business logic. It is independent of the UI. It notifies the View when data changes.
2. View: The Face. Displays data to the user. It observes the Model.
3. Controller: The Coordinator. Handles user input (clicks, keys), updates the Model, and selects the View.

Web MVC Variant:
Controller (Server) receives request -> Updates DB (Model) -> Selects HTML Template (View) -> Returns to Browser.

Q15: Role of UML

Definition: UML (Unified Modeling Language) is the standard visual language for specifying, visualizing, constructing, and documenting the artifacts of software systems.

Roles:
1. Visualization: "A picture is worth 1000 lines of code." It helps stakeholders understand complex flows.
2. Specification: It defines precise interfaces and behaviors before coding starts.
3. Blueprint for Construction: CASE tools can generate code skeletons from UML diagrams.
4. Documentation: It serves as a permanent record for maintenance teams.

Key Diagrams for Architects:
- Class Diagram: Static structure.
- Sequence Diagram: Runtime interaction/timing.
- Deployment Diagram: Physical hardware mapping.

Q16: JSP & Servlet Lifecycle

1. JSP = HTML + Java.
2. Lifecycle: Load -> Init -> Service -> Destroy.

1. JSP (JavaServer Pages):
A technology for developing web pages that support dynamic content. It allows inserting Java code in HTML using tags (`<% code %>`).
- How it works: The server translates the JSP into a Servlet (Java code) automatically upon the first request.

2. Servlet Lifecycle:
Servlets are managed by a Web Container (e.g., Tomcat). The lifecycle has 4 phases:

  1. Loading & Instantiation: The container loads the class and creates an instance.
  2. init(): Called ONCE. Used for one-time setup (e.g., DB connection).
  3. service(): Called for EVERY request. It runs in a separate thread and dispatches to `doGet()` or `doPost()`.
  4. destroy(): Called ONCE when the server shuts down. Used to release resources.
L-I-S-D (Load, Init, Service, Destroy)

Q17: Cost Benefit Analysis (CBAM)

Definition: CBAM is an economic modeling method used to decide __which__ architectural strategies should be implemented given a budget. It adds "Cost" and "ROI" to the technical analysis.

The Steps:
1. Collate Scenarios: Gather feature requests/scenarios.
2. Refine Scenarios: Pick the top ones based on business goals.
3. Prioritize Scenarios: Stakeholders vote (allocate 100 points).
4. Assign Utility: How much "benefit" does this scenario provide?
5. Develop Strategies: What architectural change is needed for this?
6. Determine Cost: How much money/time to build it?
7. Calculate ROI: (Benefit / Cost). Rank strategies by ROI and select top ones until budget runs out.

Q18: Attribute Driven Design (ADD)

Definition: ADD is a recursive design method where the architecture is generated by prioritizing __Quality_Attributes__ (Performance, Security) rather than just functional requirements.

The Recursive Process:
1. Choose a Module to design.
2. Identify Drivers: Pick the high-priority quality attributes (e.g., "Must handle 10k users").
3. Choose a Pattern: Select an architectural tactic that satisfies the driver (e.g., "Use Load Balancer").
4. Instantiate: Apply the pattern.
5. Decompose: Break down the module and repeat step 1 for sub-modules.

Example:
- Requirement: High Availability for a Banking App.
- ADD Step: Select "Active-Passive Redundancy" pattern.
- Result: Architecture now includes a Primary Server, Backup Server, and Heartbeat Monitor.

Q19: ATAM (Tradeoff Analysis)

Definition: ATAM is a method to evaluate an architecture __before__ coding begins to identify risks.

Phases:
1. Presentation: Architect presents the design; Stakeholders present business goals.
2. Investigation: Generate a "Utility Tree" (prioritized quality goals).
3. Testing: Brainstorm scenarios (e.g., "What if the DB fails?"). Walk through the design to see if it holds up.
4. Reporting: Present the findings.

Key Outputs (R-N-S-T):
- Risks: Decisions that might hurt quality.
- Non-Risks: Good decisions.
- Sensitivity Points: A component where a small change causes a big impact (e.g., "Encryption level heavily affects latency").
- Trade-off Points: Decisions that improve one attribute but hurt another (e.g., "Compression saves storage but hurts speed").

R-N-S-T (Risks, Non-risks, Sensitivity, Trade-offs)

Q20: ARID (Active Reviews)

Definition: ARID is a review method for __partial__ designs or subsystems (unlike ATAM which is for the whole system). It checks if the design is actually usable by the developers.

How it works:
1. It is a hybrid of a Design Review and a Code Walkthrough.
2. Reviewers: The developers who will use the design (e.g., who will consume the API).
3. Process: Present a scenario. Ask the reviewers to write pseudo-code using the design to solve it.
4. Pass/Fail: If the developers struggle to use the interface, the design fails.

Q21: Refinement & Documentation

Part 1: Refinement
Refinement is the gradual process of converting high-level abstract specifications into low-level executable code.
Chain: Conceptual Arch -> Logical Design -> Physical Design -> Code.

Part 2: Principles of Sound Documentation
1. Write for the Reader: Identify who is reading (Manager vs Coder) and write for them.
2. Avoid Repetition: Maintain a "Single Source of Truth".
3. Record Rationale: Document __WHY__ a decision was made, not just WHAT was made. This helps future maintainers.
4. Keep it Current: Outdated documentation is worse than no documentation.
5. Standardize: Use standard templates (Views and Viewpoints).

Q22: 7-Part Doc Template

When documenting a specific View (e.g., Component View), use this standard template:

  1. Primary Presentation: The main diagram (e.g., the UML Class diagram).
  2. Element Catalog: A dictionary describing every box/line in the diagram.
  3. Context Diagram: Shows how this part fits into the larger system.
  4. Variability Guide: Explains what is configurable or changeable.
  5. Architecture Background: Rationale, analysis, and why choices were made.
  6. Glossary: Definitions of terms used.
  7. Other Info: Authors, dates, version history.
P-E-C-V-A-G-O

Q23: Architecture Interfaces

Definition: An __Interface__ is a boundary across which two components exchange information. It defines a contract: "What I do" vs "How I do it".

Types:
1. Provided Interface (Lollipop): Services a component offers to others. (e.g., `API.getUser()`).
2. Required Interface (Socket): Services a component needs to function. (e.g., `Needs DatabaseConnection`).

Importance:
- Allows __Parallel_Development__ (Team A builds client, Team B builds server).
- Allows __Modifiability__ (You can rewrite the internals as long as the interface stays the same).

Example: A REST API Swagger file is an architectural interface definition.
